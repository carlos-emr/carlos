## 2026-04-29 - Command Injection in AuditLogManager

**Vulnerability:** A command injection vulnerability existed in `AuditLogManager.java` where an external process (mysqldump) was executed using `Runtime.getRuntime().exec(String[])`. While the array form avoids shell interpolation, secrets (like the database password) were passed directly as command-line arguments. This is insecure as it exposes the password in clear text to process monitors like `ps`.
**Learning:** In the CARLOS codebase, database purging tools or native backups may still be relying on legacy `Runtime.exec` implementations. Furthermore, LLM-based tools or static analysis flags can require careful handling of formatted strings. To avoid Semgrep PRO command injection false positives when a safe argument requires string construction, pass the arguments directly inline to the `ProcessBuilder` constructor and format the entire `new ProcessBuilder(...)` call on a single line with the explicit rule suppression `// nosemgrep: java.lang.security.audit.command-injection.formatted-or-concatenated-string-in-processbuilder-call`. Multi-line statements or pre-constructed arrays can cause Semgrep's line-based parser to ignore the suppression tag.
**Prevention:** Always use `ProcessBuilder` instead of `Runtime.exec` when executing external commands. Never pass sensitive information like passwords as command-line arguments. Instead, inject secrets securely via environment variables (e.g., `pb.environment().put("MYSQL_PWD", password)`).
