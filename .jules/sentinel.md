## 2024-05-24 - [SQL Injection and Command Injection]
**Vulnerability:** We found raw string concatenation used in `entityManager.createQuery(...)`, specifically with the `IN` clause for `ids` (e.g., `"update Appointment set status=?1 where id in (" + idClean.toString() + ")"` in `OscarAppointmentDaoImpl.java`). We also found command injection vulnerability with `Runtime.getRuntime().exec` passing unsanitized arguments, or arguments that include raw strings, such as `vars[2] = "--password=" + password;` which can expose database credentials in the process list and could be injected into depending on the properties file parsing. `PGPEncrypt.java` seems to pass the `dir` variable from user input? No, wait, in PGPEncrypt it's `rt.exec(cmd, env, dir)` and `cmd` is constructed using `PGP_BIN`, `PGP_CMD`, `srcFile` and `PGP_KEY`. Wait, if `srcFile` comes from user input, this is a command injection vulnerability. Wait, looking at `AuditLogManager.java`, it passes `--password=xxxx` on the command line. This exposes the DB password to any user running `ps aux` on the server!

**Learning:** `Runtime.getRuntime().exec` with sensitive data like passwords should be avoided because it exposes credentials to process monitors. When dealing with database credentials in commands like `mysqldump`, we should try to pass the password via environment variables (like `MYSQL_PWD`) or temporary configuration files to avoid exposing them in the command line arguments. For HQL `IN` clauses, Hibernate supports passing a `List` directly to the named/positional parameter, avoiding the need for raw string concatenation.

**Prevention:** Use parameterized queries instead of string concatenation for ALL query parameters, including `IN` clauses. Avoid passing sensitive credentials in command line arguments when invoking external processes.
